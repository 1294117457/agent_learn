## 一、ID-AIDemo 独立开发阶段

### 目标
将现有只有一个对话脚本的 `src/index.ts` 改造为完整的 RAG Agent HTTP 服务，基于 `docs/加分文件/` 下的现有政策文件作为知识库冷启动数据。

### 最终文件结构

```
ID-AIDemo/
├── src/
│   ├── index.ts              ← Express 服务入口（改造现有文件）
│   ├── routes/
│   │   ├── chat.ts           ← POST /chat/send、POST /chat/clear
│   │   └── knowledge.ts      ← GET /knowledge/list、POST /knowledge/upload、DELETE /knowledge/:id
│   ├── services/
│   │   ├── docParser.ts      ← PDF/Word/Excel/MD 文本提取
│   │   ├── embeddings.ts     ← Qwen text-embedding-v3 调用封装
│   │   ├── vectorStore.ts    ← SQLite 存储 + 余弦相似度检索
│   │   └── ragChain.ts       ← LangChain RAG 流水线（检索 → 拼 Prompt → Qwen3）
│   ├── db/
│   │   └── init.ts           ← SQLite 建表（knowledge_chunks、conversations）
│   └── seed/
│       └── seedKnowledge.ts  ← 启动时自动加载 docs/加分文件/ 下所有文档
├── docs/
│   └── 加分文件/             ← 现有政策文件（只读，不修改）
├── .env                      ← QWEN3_API_KEY、INTERNAL_TOKEN、PORT 等
├── package.json
└── tsconfig.json
```

---

### 启动流程

```
pnpm start
  1. db/init.ts    → 初始化 SQLite（knowledge_chunks 表不存在时建表）
  2. seed/seedKnowledge.ts → 扫描 docs/加分文件/，解析所有文件，
                             切片 → 调用 embedding → 存入 SQLite
                             （已存在的同名文件跳过，避免重复）
  3. Express 启动  → 监听 :3001
```

### 知识库文件处理策略

| 文件 | 解析方式 | 说明 |
|------|----------|------|
| `加分政策.md` | 直接读取文本 | 主要规则文档，切片粒度可大一些（~500字/块）|
| `附件1...细则.docx` / `附件3...特殊学术专长.doc` | `mammoth` 库转 HTML 再转纯文本 | 包含详细资格条件 |
| `附件2...操作办法.pdf` | `pdf-parse` | 综合成绩计算方法 |
| `附件3...换算关系表.xlsx` | `xlsx` 库按行转为文本 | "绩点 X → 学业成绩 Y分" 映射表，转成可读文本行 |
| `附件5...登记表.xlsx` | `xlsx` 库按行转为文本 | 加分登记表表头即为字段说明 |

### SQLite 表结构

```sql
-- 知识库文本块
CREATE TABLE IF NOT EXISTS knowledge_chunks (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  source_file TEXT    NOT NULL,
  chunk_index INTEGER NOT NULL,
  content     TEXT    NOT NULL,
  embedding   TEXT    NOT NULL,   -- JSON 浮点数组
  created_at  INTEGER NOT NULL DEFAULT (unixepoch())
);

-- 对话历史（按 session_id 隔离多用户）
CREATE TABLE IF NOT EXISTS conversations (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id TEXT    NOT NULL,   -- userId 或临时 UUID
  role       TEXT    NOT NULL,   -- 'user' | 'assistant'
  content    TEXT    NOT NULL,
  created_at INTEGER NOT NULL DEFAULT (unixepoch())
);
```

---

### RAG 对话流程

```
POST /chat/send  { sessionId, message }
  │
  ├─ 1. 对 message 调用 embeddings.ts 得到向量
  ├─ 2. vectorStore.ts：对 knowledge_chunks 全表加载（条数少时）
  │      计算余弦相似度，取 top-5 相关块
  ├─ 3. 从 conversations 读取该 sessionId 最近 6 条历史
  ├─ 4. 组装 Prompt：
  │      SystemMessage: "你是厦门大学信息学院推免助手...以下是相关政策：[top5 chunks]"
  │      历史消息（HumanMessage / AIMessage 交替）
  │      HumanMessage: message
  ├─ 5. 调用 Qwen3 得到 reply
  ├─ 6. 将 user + assistant 消息写入 conversations 表
  └─ 7. 返回 { code: 200, data: { reply } }
```

---

### 对话接口设计（独立阶段）

```
POST /chat/send
  Body: { "sessionId": "string", "message": "string" }
  Response: { "code": 200, "data": { "reply": "string" } }

POST /chat/clear
  Body: { "sessionId": "string" }
  Response: { "code": 200, "data": null }

GET /knowledge/list
  Response: { "code": 200, "data": [ { "id", "sourceFile", "chunkCount" } ] }

POST /knowledge/upload
  Form: file (pdf/docx/doc/xlsx/md)
  Response: { "code": 200, "data": { "chunkCount": N } }

DELETE /knowledge/:id
  Response: { "code": 200, "data": null }
```

## 二、Agent 与后端交互方案

### 方案概述

在独立阶段完成后，将 ID-AIDemo 以**独立微服务**方式运行，idbackend 作为唯一入口对前端提供服务，内部通过 HTTP 调用 ID-AIDemo。

```
前端（Vue）
    │  POST /api/chat/send（带 JWT）
    ▼
idbackend（Java, :8080）
    │  验证 JWT → 提取 userId
    │  POST http://localhost:3001/chat/send（带 X-Internal-Token）
    ▼
ID-AIDemo（Node.js, :3001）
    │  验证 X-Internal-Token → 执行 RAG → 返回结果
    ▼
idbackend 返回给前端
```

---

### 三种交互方案对比

#### 方案 A：HTTP 同步调用（推荐）

idbackend 的 `AIAgentService.java` 直接用 `RestTemplate` 或 `WebClient` 调 ID-AIDemo。

**优点**：
- 实现最简单，无额外依赖
- 响应链路短，易于调试
- 已有的 `ChatController` 改动最小

**缺点**：
- 同步阻塞（可用 `WebClient` 异步化缓解）
- Agent 超时会拖慢接口响应

**推荐配置**：
- 超时：连接 2s，读取 30s（Qwen3 生成可能较慢）
- 失败行为：catch 后返回降级消息（见第三节）

---

#### 方案 B：异步消息队列（不推荐，过重）

引入 RabbitMQ / Redis Stream，idbackend 发消息，ID-AIDemo 消费并回调。

适合 AI 任务耗时极长（>1分钟）的场景，对于对话问答不必要。当前项目不建议引入。

---

#### 方案 C：前端直连 ID-AIDemo（不推荐）

前端跳过 idbackend 直接调 ID-AIDemo，省去一层转发。

问题：ID-AIDemo 需要自己做 JWT 验证，与 Java 侧的 token 逻辑重复，维护成本高。用户已明确要"前端发给后端，后端和 agent 交互"，排除此方案。

---

### 推荐方案 A 的 idbackend 改动

**新增文件**：`service/functionService/AIAgentService.java`

```java
// 核心逻辑示意（非完整代码）
public String chat(String message, Long userId) {
    try {
        Map<String, Object> body = Map.of(
            "sessionId", "user_" + userId,
            "message", message
        );
        ResponseEntity<Map> resp = restTemplate.postForEntity(
            agentBaseUrl + "/chat/send",
            buildRequest(body),    // 含 X-Internal-Token header
            Map.class
        );
        return (String) ((Map) resp.getBody().get("data")).get("reply");
    } catch (Exception e) {
        // 降级处理（见第三节）
        return null;
    }
}
```

**改动**：`ChatController.java` 将 `QianFanService` 替换为 `AIAgentService`，逻辑不变。

## 三、宕机容灾：Agent 宕机不影响原系统

### 核心原则

所有 AI 功能走独立 Controller/Service 分支，原有业务功能完全不依赖 AI 服务。

### 降级策略

| 场景 | Agent 正常 | Agent 宕机/超时 |
|------|-----------|----------------|
| 聊天对话 | 返回 RAG 回答 | 返回"AI服务暂时不可用，请稍后重试" |
| 证明材料分析 | 返回加分推荐 | 返回空列表 + 提示"AI分析暂时不可用，请手动选择" |
| 申请生成 | 返回填好的表单 | 返回空表单，提示学生手动填写 |
| 知识库上传 | 文件入库 + 向量化 | 提示"文件已保存，AI处理队列暂时不可用" |

**前端处理**：所有 AI 接口的返回结构统一为：

```json
{ "code": 200, "msg": "成功", "data": { ... }, "aiAvailable": true }
// 降级时：
{ "code": 200, "msg": "AI服务暂时不可用", "data": null, "aiAvailable": false }
```

前端根据 `aiAvailable` 字段决定显示 AI 结果还是提示语，不报错、不崩溃。

### idbackend 超时保护

```java
// application.yml
ai-agent:
  base-url: http://localhost:3001
  internal-token: ${AI_INTERNAL_TOKEN}
  timeout:
    connect: 2000   # 连接超时 2s
    read: 30000     # 读取超时 30s（Qwen3 生成可能较慢）
```

超时抛出 `ResourceAccessException` → 统一 catch → 返回降级 `ResultVo`。

### 简单熔断（可选）

在 `AIAgentService` 中记录最近连续失败次数：连续 5 次失败后，进入"熔断"状态，直接返回降级消息，不再尝试调用，每 60s 自动恢复探测。避免 Agent 宕机时每个请求都等 30s 超时。

## 四、实施步骤（仅改 ID-AIDemo 阶段）

```

Step 9  routes/knowledge.ts：实现知识库管理接口

```
