## 一、代码 Bug（仍需修复）

### Bug 1：`package.json` 缺少 `zod`

`applyGraph.ts` 第 7 行和 `mainGraph.ts` 第 9 行都有 `import { z } from 'zod'`，但 `package.json` 的 dependencies 里没有 `zod`。

**操作**：`npm install zod`

---

### Bug 2：`src/db/mysql.ts` 第 2 行的 import 来源即将失效

```typescript
import type { ScoreTemplate } from '../services/analyzeChain.js'
```

`mysql.ts` 从 `analyzeChain.ts` 导入 `ScoreTemplate` 类型。但 `types/scoreTemplate.ts` 里已经有一模一样的定义。如果将来删掉 `analyzeChain.ts`，这行就会报错。

**操作**：改为 `import type { ScoreTemplate } from '../types/scoreTemplate.js'`

这样 `mysql.ts` 就不再依赖 `analyzeChain.ts`，两个文件可以独立存在。

---

### Bug 3：`analyzeChain.ts` 里有重复的类型定义

`analyzeChain.ts` 第 9-24 行定义了 `TemplateRule`、`ScoreTemplate`，和 `types/scoreTemplate.ts` 完全重复。而且 `analyze.ts` 路由第 10 行是 `import ... type ScoreTemplate from '../services/analyzeChain.js'`。

**操作**：
- `analyzeChain.ts` 删掉第 9-24 行的类型定义，改为在顶部加 `import type { ScoreTemplate } from '../types/scoreTemplate.js'`
- `analyze.ts` 路由第 7-11 行改为：
  ```typescript
  import { analyzeCertificate, generateApplicationRemark } from '../services/analyzeChain.js'
  import type { ScoreTemplate } from '../types/scoreTemplate.js'
  ```

这样所有地方都从 `types/scoreTemplate.ts` 这一个地方拿类型，不会有重复。

## 二、后端交互检查

### 2.1 新旧接口对比

| 接口 | 旧路由 | 新路由 | idbackend 目前调用的 |
|------|--------|--------|---------------------|
| 政策咨询 | `POST /chat/send` | `POST /agent/chat` | `/chat/send` |
| 流式咨询 | `POST /chat/stream` | 暂无 | `/chat/stream` |
| 证书分析 | `POST /analyze/certificate` | `POST /agent/chat` + file | `/analyze/certificate` |
| 申请生成 | `POST /analyze/generate` | 无替代 | `/analyze/generate` |
| interrupt恢复 | 无 | `POST /agent/resume` | 无 |

### 2.2 旧接口：idbackend 仍在调用，不能删

**`POST /analyze/certificate`**

idbackend 调用方式（multipart/form-data）：
```
file: PDF文件
templates: JSON字符串 ScoreTemplate[]
```

返回格式：
```json
{
  "code": 200,
  "data": {
    "certificateText": "...",
    "suggestions": [{ "templateId": 1, "ruleId": 2, "estimatedScore": 4.5, ... }]
  }
}
```

这个接口由 `routes/analyze.ts` → `analyzeChain.ts` 提供。新的 `agent/chat` 虽然能做同样的事，但**返回格式不同**（多了 `intent`、`reply` 字段，`suggestions` 的来源是 `checkResults` 解析），idbackend 目前的代码不认识新格式。

**结论：`analyze.ts` + `analyzeChain.ts` 必须保留，直到 idbackend 改为调用新接口。**

**`POST /analyze/generate`**

idbackend 调用方式（JSON）：
```json
{
  "certificateText": "...",
  "selectedTemplateId": 1,
  "selectedRuleId": 2,
  "template": { ... }
}
```

这个功能新路由完全没有替代。它是一个简单的一次性 LLM 调用（生成申请备注文本），不需要 LangGraph。

**结论：`analyzeChain.ts` 的 `generateApplicationRemark()` 必须保留。**

**`POST /chat/send` 和 `POST /chat/stream`**

前端/idbackend 调用方式（JSON）：
```json
{ "sessionId": "xxx", "message": "挑战杯能加多少分？" }
```

新的 `/agent/chat` 功能上完全可以替代。但返回格式有差异：

| 字段 | 旧 `/chat/send` 返回 | 新 `/agent/chat` 返回 |
|------|---------------------|----------------------|
| `data.reply` | `"回答文本"` | `"回答文本"` |
| `data.intent` | 无 | `"consult"` / `"apply"` / `"insufficient"` |
| `data.suggestions` | 无 | `[]` 或匹配结果数组 |
| `data.interrupted` | 无 | `true`（仅 interrupt 时） |
| `aiAvailable` | `true/false` | 无 |

**结论：旧的 `chat.ts` + `ragChain.ts` 暂时保留，等前端适配新返回格式后再删。**

### 2.3 新接口参数详解（供 idbackend/前端对接用）

**`POST /agent/chat`**（multipart/form-data）

请求参数：
```
message    : string   必填（或 file 二选一）
sessionId  : string   必填，会话ID
file       : File     可选，PDF/DOCX 等材料
templates  : string   可选，JSON 字符串的 ScoreTemplate[]
```

正常返回：
```json
{ "code": 200, "msg": "成功", "data": { "reply": "...", "intent": "consult", "suggestions": [] } }
```

interrupt 返回：
```json
{ "code": 200, "msg": "需要补充信息", "data": { "interrupted": true, "question": "还缺：奖项等级、时间" } }
```

**`POST /agent/resume`**（JSON）

请求参数：
```json
{ "sessionId": "和之前一致", "supplement": "用户补充的内容" }
```

返回：
```json
{ "code": 200, "msg": "成功", "data": { "reply": "...", "intent": "apply" } }
```
