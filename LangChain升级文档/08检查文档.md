### 1.3 逐项对比

#### F1 政策咨询 → 新 consultSubgraph

| 方面 | 旧（ragChain） | 新（consultSubgraph） | 是否偏移 |
|------|--------------|---------------------|---------|
| 知识检索 | embedding → Top-5 余弦 | embedding → Top-5 余弦 | ✅ 一致 |
| System Role | getSystemRole() 动态读取 | getSystemRole() 动态读取 | ✅ 一致 |
| 对话历史 | SQLite 持久化，最近 6 条 | MemorySaver 内存，重启丢失 | ⚠️ 降级 |
| 历史保存 | saveMessage → SQLite | 无（不写 conversations 表） | ⚠️ 缺失 |
| 会话清除 | /chat/clear 删 SQLite 记录 | 无替代 API | ⚠️ 缺失 |

**结论：consult 核心逻辑一致，但对话历史管理从 SQLite 持久化降级为纯内存。服务重启后所有对话上下文丢失。**

#### F2 流式咨询 → 新 Agent 无替代

| 方面 | 旧 | 新 |
|------|-----|-----|
| 流式输出 | ✅ chatModel.stream() → SSE | ❌ 无流式支持 |

**结论：新 /agent/chat 只有 invoke（等整个图跑完才返回），没有流式。如果 consultSubgraph 的 answerNode 要 30 秒才返回，前端会白屏等 30 秒。**

#### F4 证书分析 → 新 applySubgraph

| 方面 | 旧（analyzeChain） | 新（applySubgraph） | 是否偏移 |
|------|-------------------|---------------------|---------|
| PDF 解析 | route 层 parseFile | route 层 parseFile | ✅ 一致 |
| RAG 检索 | embedding → search | embedding → search | ✅ 一致 |
| 模板匹配 | LLM → 手动 JSON.parse | LLM → withStructuredOutput(zod) | ✅ 升级（更稳定） |
| 返回格式 | `{ certificateText, suggestions }` | `{ reply, intent, suggestions }` | ⚠️ 格式不同 |

**新的 /agent/chat 返回中没有 `certificateText` 字段**，而旧的 idbackend 流程里，`certificateText` 需要传给第二步 `/analyze/generate`。

#### F5 申请生成 → 新 Agent 完全没有替代

旧流程是 2 步：
```
Step 1: POST /analyze/certificate → { certificateText, suggestions }
Step 2: 学生选择一项 → POST /analyze/generate → { templateName, applyScore, ruleId, remark }
```

新 Agent 是 1 步到底：
```
POST /agent/chat (带文件) → classifyNode → applySubgraph → 返回 reply + suggestions
```

新 Agent 没有第 2 步（根据学生选择生成具体的申请备注和表单预填数据）。`analyzeChain.ts` 的 `generateApplicationRemark()` 在新 LangGraph 中没有对应节点。

**结论：这是最大的功能缺口。新 Agent 只做了"分析推荐"，没有做"生成申请"。**

#### F6、F7 知识库管理 / AI 配置 → 完全不受影响

这两个功能的路由和代码完全没动，不存在偏移。

### 1.4 偏移汇总

| 编号 | 问题 | 严重程度 | 说明 |
|------|------|---------|------|
| D1 | 对话历史不持久化 | 中 | MemorySaver 是纯内存，重启丢失 |
| D2 | 无流式输出 | 中 | 前端等待时间长，体验差 |
| D3 | 无"申请生成"步骤 | 高 | 学生选完推荐后无法生成 remark 和表单数据 |
| D4 | 返回格式变了 | 低 | 新返回多了 intent/reply，少了 certificateText |
| D5 | 无会话清除 API | 低 | MemorySaver 无法针对单个 thread 清除 |

## 二、代码 Bug

### Bug 1：interrupt 检测方式可能在 v1.x 失效

`src/routes/agent.ts` 第 81-94 行用 try/catch 捕获 interrupt：

```typescript
try {
    const result = await invokeAgent({...})
    res.json({ code: 200, msg: '成功', data: result })
} catch (err: any) {
    if (err?.interrupts?.length > 0 || err?.name === 'GraphInterrupt') {
        // 处理 interrupt
    }
}
```

在 `@langchain/langgraph` v1.x 中，`interrupt()` 不一定会让 `invoke()` 抛出异常。更可能的行为是：`invoke()` 正常返回（返回中断时的部分 state），interrupt 信息存储在 checkpoint 中。

如果 `invoke()` 不抛，catch 永远不会执行，路由会返回一个 reply 为空、intent 为 `insufficient` 的"成功"响应，前端无法知道需要补充信息。

**推荐修复**：改用 `app.getState()` 检测 interrupt：

```typescript
// mainGraph.ts 中
export async function invokeAgent(input: AgentInput) {
  const config = { configurable: { thread_id: input.sessionId } }
  
  const result = await app.invoke({
    messages: [new HumanMessage(input.userInput)],
    documentText: input.documentText ?? '',
    templates: input.templates ?? [],
  }, config)

  // 检查是否被 interrupt 中断
  const snapshot = await app.getState(config)
  const interruptTask = snapshot.tasks?.find(
    (t: any) => t.interrupts && t.interrupts.length > 0
  )

  if (interruptTask) {
    return {
      interrupted: true,
      question: String(interruptTask.interrupts[0].value),
      intent: result.intent,
    }
  }

  // 正常完成
  const lastAI = result.messages
    .filter((m: any) => m._getType() === 'ai')
    .at(-1)

  return {
    reply: String(lastAI?.content ?? ''),
    intent: result.intent,
    interrupted: false,
    suggestions: result.checkResults
      .map((r: string) => { try { return JSON.parse(r) } catch { return null } })
      .filter(Boolean),
  }
}
```

然后 `routes/agent.ts` 的 /chat 路由改为：

```typescript
router.post('/chat', upload.single('file'), async (req, res) => {
  // ... 前面解析参数不变 ...

  try {
    const result = await invokeAgent({ userInput, documentText, templates, sessionId })

    if (result.interrupted) {
      res.json({
        code: 200,
        msg: '需要补充信息',
        data: { interrupted: true, question: result.question, intent: result.intent }
      })
    } else {
      res.json({ code: 200, msg: '成功', data: result })
    }
  } catch (err: any) {
    console.error('[agent/chat] error:', err)
    res.json({ code: 500, msg: `处理失败: ${String(err)}`, data: null })
  }
})
```

#### (A) AIAgentService.java 新增 2 个方法

```java
/**
 * 调用新 LangGraph Agent 统一入口
 * POST /agent/chat (multipart/form-data)
 */
public String agentChat(String message, String sessionId, 
                        MultipartFile file, List<?> templates) {
    try {
        MultipartBody.Builder bodyBuilder = new MultipartBody.Builder()
            .setType(MultipartBody.FORM)
            .addFormDataPart("message", message)
            .addFormDataPart("sessionId", sessionId);

        if (templates != null && !templates.isEmpty()) {
            bodyBuilder.addFormDataPart("templates", JSON.toJSONString(templates));
        }

        if (file != null && !file.isEmpty()) {
            String filename = file.getOriginalFilename() != null 
                ? file.getOriginalFilename() : "upload";
            RequestBody fileBody = RequestBody.create(file.getBytes(),
                MediaType.parse(file.getContentType() != null 
                    ? file.getContentType() : "application/octet-stream"));
            bodyBuilder.addFormDataPart("file", filename, fileBody);
        }

        Request req = new Request.Builder()
            .url(baseUrl + "/agent/chat")
            .post(bodyBuilder.build())
            .build();
        try (Response resp = http.newCall(req).execute()) {
            return resp.body() != null ? resp.body().string() : "{}";
        }
    } catch (Exception e) {
        log.error("[AIAgent] agentChat 调用失败", e);
        return "{\"code\":500,\"msg\":\"Agent 不可达\",\"data\":null}";
    }
}

/**
 * 恢复 interrupt（用户补充信息后调用）
 * POST /agent/resume (JSON)
 */
public String agentResume(String sessionId, String supplement) {
    Map<String, String> body = new HashMap<>();
    body.put("sessionId", sessionId);
    body.put("supplement", supplement);
    try {
        return postJson("/agent/resume", body);
    } catch (Exception e) {
        log.error("[AIAgent] agentResume 调用失败", e);
        return "{\"code\":500,\"msg\":\"Agent 不可达\",\"data\":null}";
    }
}
```

        // 从 DB 查模板

        String sessionId = userIdInt != null ? "user_" + userIdInt : "anonymous";

        JSONObject json = JSON.parseObject(raw);

        if (json.getInteger("code") == 200) {
            return ResultVo.success("成功", json.get("data"));
        }
        return ResultVo.error("恢复失败: " + json.getString("msg"));
    }

    @Data

### 3.4 前端适配要点

前端需要处理 3 种响应：

```
1. 正常回复（consult 或 apply）:
   { code: 200, data: { reply: "...", intent: "consult", suggestions: [] } }

2. 需要补充信息（interrupted）:
   { code: 200, data: { interrupted: true, question: "还缺：奖项等级、时间" } }
   → 前端显示 question，用户输入后调 POST /api/agent/resume

3. 错误:
   { code: 500, msg: "..." }
```

## 五、整体建议

### 当前策略：新旧并行（推荐）

```
前端学生端
  ├── 旧入口：/api/chat/stream（政策咨询，用旧 ragChain，有流式）
  ├── 旧入口：/api/ai/analyze/certificate → /generate（证书分析 2 步流程）
  └── 新入口：/api/agent/chat + /resume（LangGraph 统一智能路由）

前端管理端
  ├── /api/knowledge/*（知识库管理，不变）
  └── /api/ai-config（AI 配置，不变）
```

现阶段的正确做法是：
1. **旧接口全部保留**，idbackend 不改任何现有代码
2. **新增** AIAgentService 的 `agentChat()` / `agentResume()` 方法
3. **新增** AIAgentController 暴露 /api/agent/* 给前端
4. 前端**新建**一个"智能助手"页面对接新接口
5. 等新接口稳定后，再逐步废弃旧接口

### 后续需要补齐的 Agent 功能

| 优先级 | 功能 | 说明 |
|--------|------|------|
| P0 | 修复 interrupt 检测 | 改用 getState() 而非 try/catch（见 Bug 1） |
| P1 | Agent 端补 /analyze/generate 能力 | 在 applySubgraph 中加 generateNode，或保留旧接口不变 |
| P2 | 流式输出 | 新 Agent 加 stream 支持（LangGraph 支持 `.stream()` 和 `.streamEvents()`） |
