### Step 4：`src/agents/subgraphs/consultGraph.ts`（核心改动①）

**来源**：你的 `21day/src/subgraphs/consultGraph.ts`

**这是改动最大的地方**——把 mock 检索替换为真实的 `embeddings.ts` + `vectorStore.ts`。

**与 demo 的区别**：

| demo 中 | 项目中需要调整 |
|---------|--------------|
| `retrieveNode` 返回硬编码字符串 | 调用 `getEmbedding()` + `searchSimilar()` 做真实向量检索 |
| `answerNode` 用固定 system prompt | 用 `getSystemRole()` 读取动态配置的 system prompt |
| `model` 从 `@/llm` 导入 | 用 `aiConfig.ts` 每次创建 |

**实现要点**：

```typescript
import { getEmbedding } from '../../services/embeddings.js'      // ← 真实 embedding
import { searchSimilar } from '../../services/vectorStore.js'     // ← 真实向量检索
import { getApiKey, getBaseUrl, getChatModel, getSystemRole } from '../../services/aiConfig.js'

// retrieveNode：把你 demo 的 mock 换成真实调用
async function retrieveNode(state: ConsultStateType): Promise<Partial<ConsultStateType>> {
  const userMsg = state.messages.filter(m => m instanceof HumanMessage).at(-1)!
  const query = String(userMsg.content)

  // ← 这两行就是从 ragChain.ts 的 buildContext() 搬过来的
  const queryVec = await getEmbedding(query)
  const chunks = searchSimilar(queryVec, 5)

  const context = chunks.length > 0
    ? chunks.map((c, i) => `[${i + 1}]（相似度${c.similarity.toFixed(2)}）${c.content}`).join('\n\n')
    : '（知识库暂无相关内容）'

  return { retrievedContext: context }
}

// answerNode：基本和 demo 一样，system prompt 改为动态读取
async function answerNode(state: ConsultStateType): Promise<Partial<ConsultStateType>> {
  const model = createModel()   // ← 用 aiConfig 创建
  const userMsg = state.messages.filter(m => m instanceof HumanMessage).at(-1)!

  const reply = await model.invoke([
    new SystemMessage(`${getSystemRole()}\n\n【知识库检索结果】\n${state.retrievedContext}`),
    new HumanMessage(String(userMsg.content)),
  ])
  return { answerDraft: String(reply.content), messages: [reply] }
}
```

**本质上就是**：你 demo 的 `consultGraph.ts` + 从 `ragChain.ts` 搬来的 `buildContext()` 逻辑。

**工作量**：约 30 分钟。

### Step 5：`src/agents/subgraphs/applyGraph.ts`（核心改动②）

**来源**：你的 `21day/src/subgraphs/applyGraph.ts`

**需要把 mock 检查替换为真实的模板匹配逻辑。**

**与 demo 的区别**：

| demo 中 | 项目中需要调整 |
|---------|--------------|
| 三个 mock prompt（policy_match/format_check/time_validity） | 真实检查：向量检索政策 + 模板匹配 + 时间校验 |
| `summarizeNode` 简单总结 | 生成 `AnalyzeSuggestion[]` 结构化结果（对齐 idbackend DTO） |
| 无模板数据 | 从 `state.templates` 读取 `ScoreTemplate[]` |

**实现策略**：保留你 demo 的 `Send` 并行架构，但把三个检查任务改为更实际的内容。

**方案A（推荐，简洁）**：不用 Send 并行，因为真实场景中"检索策略→模板匹配→生成建议"是有依赖的顺序流程。

```
applyGraph 节点流：
  START → fetchPolicy → matchTemplates → generateSuggestion → END
```

```typescript
import { getEmbedding } from '../../services/embeddings.js'
import { searchSimilar } from '../../services/vectorStore.js'

// 节点1：检索相关政策（从 analyzeChain.ts 搬来）
async function fetchPolicyNode(state: ApplyStateType): Promise<Partial<ApplyStateType>> {
  const queryVec = await getEmbedding(state.documentText.slice(0, 512))
  const chunks = searchSimilar(queryVec, 5)
  const policyContext = chunks.length > 0
    ? chunks.map((c, i) => `[${i + 1}] ${c.content}`).join('\n\n')
    : '（知识库暂无相关政策）'
  return { policyContext }
}

// 节点2：模板匹配（用 withStructuredOutput，替代 analyzeChain.ts 的手动 JSON.parse）
async function matchTemplatesNode(state: ApplyStateType): Promise<Partial<ApplyStateType>> {
  const SuggestionSchema = z.array(z.object({
    templateId: z.number(),
    templateName: z.string(),
    ruleId: z.number(),
    ruleName: z.string(),
    estimatedScore: z.number(),
    reason: z.string(),
  }))

  const model = createModel().withStructuredOutput(SuggestionSchema)

  const templatesForPrompt = state.templates.map(t => ({
    id: t.id,
    templateName: t.templateName,
    rules: t.rules.map(r => ({ id: r.id, ruleName: r.ruleName, ruleScore: r.ruleScore })),
  }))

  const result = await model.invoke([
    new SystemMessage('你是推免加分审核专家。分析材料，返回匹配的加分项。无匹配返回空数组。'),
    new HumanMessage(`材料：\n${state.documentText.slice(0, 2000)}
    \n可用模板：\n${JSON.stringify(templatesForPrompt, null, 2)}
    \n参考政策：\n${state.policyContext}`),
  ])

  // result 直接是 AnalyzeSuggestion[]，不需要 JSON.parse！
  // 这就是你 Day 7 学的 withStructuredOutput 的真正价值
  return { checkResults: result.map(r => JSON.stringify(r)) }
}

// 节点3：生成最终回复
async function generateResponseNode(state: ApplyStateType): Promise<Partial<ApplyStateType>> {
  // 把匹配结果整理为用户友好的消息
  const suggestions = state.checkResults.map(r => JSON.parse(r))
  if (suggestions.length === 0) {
    return { messages: [new AIMessage('未找到匹配的加分项。请确认材料内容或补充更多信息。')] }
  }
  const summary = suggestions.map((s: any) =>
    `- ${s.templateName} / ${s.ruleName}：预计 ${s.estimatedScore} 分（${s.reason}）`
  ).join('\n')

  return { messages: [new AIMessage(`为您匹配到以下加分项：\n${summary}`)] }
}
```

**方案B（保留你 demo 的 Send 架构）**：如果你想保留并行检查，可以把三个检查改为"政策匹配检查""格式完整性检查""时间有效性检查"，每个都用真实 LLM 调用。适合想练手 Send 的场景，但会消耗 3 倍 token。

**推荐方案A**，因为真实业务中这三步是有依赖的（先检索政策才能匹配模板）。

**工作量**：约 45 分钟。

### Step 6：`src/agents/mainGraph.ts`

**来源**：你的 `21day/src/subgraphs/mainGraph.ts`，几乎照搬。

**与 demo 的区别**：

| demo 中 | 项目中需要调整 |
|---------|--------------|
| `config` 和 `checkpointer` 写在文件里 | `checkpointer` 写在文件里，`config` 从路由传入 |
| 无 export 接口函数 | 新增 `invokeAgent()` / `resumeAgent()` / `streamAgent()` 三个接口 |
| 直接 `await app.invoke()` 测试 | 不在此文件运行，由路由调用 |

**实现要点**：

```typescript
import { MemorySaver, Command } from '@langchain/langgraph'
import { classifyNode } from './nodes/classifyNode.js'
import { clarifyNode } from './nodes/clarifyNode.js'
import { consultSubgraph } from './subgraphs/consultGraph.js'
import { applySubgraph } from './subgraphs/applyGraph.js'

const checkpointer = new MemorySaver()

// ← 图的构建和你 demo 一模一样
export const mainGraph = new StateGraph(MainState)
  .addNode('classify', classifyNode)
  .addNode('ask', clarifyNode)
  .addNode('applyGraph', applySubgraph)
  .addNode('consultGraph', consultSubgraph)
  .addEdge(START, 'classify')
  .addConditionalEdges('classify', (s) => s.intent, {
    insufficient: 'ask',
    apply: 'applyGraph',
    consult: 'consultGraph',
  })
  .addEdge('ask', 'classify')
  .addEdge('applyGraph', END)
  .addEdge('consultGraph', END)

const app = mainGraph.compile({ checkpointer })

// ↓↓↓ 以下是新增的接口函数，供路由调用 ↓↓↓

export interface AgentInput {
  userInput: string
  documentText?: string
  templates?: ScoreTemplate[]
  sessionId: string
}

export async function invokeAgent(input: AgentInput) {
  const config = { configurable: { thread_id: input.sessionId } }
  const result = await app.invoke({
    messages: [new HumanMessage(input.userInput)],
    documentText: input.documentText ?? '',
    hasFile: !!input.documentText,
    templates: input.templates ?? [],
  }, config)

  const lastAI = result.messages.filter((m: any) => m._getType() === 'ai').at(-1)
  return {
    reply: String(lastAI?.content ?? ''),
    intent: result.intent,
    // 如果有匹配结果，解析为 AnalyzeSuggestion[]
    suggestions: result.checkResults
      .map((r: string) => { try { return JSON.parse(r) } catch { return null } })
      .filter(Boolean),
  }
}

export async function resumeAgent(sessionId: string, supplement: string) {
  const config = { configurable: { thread_id: sessionId } }
  const result = await app.invoke(new Command({ resume: supplement }), config)
  const lastAI = result.messages.filter((m: any) => m._getType() === 'ai').at(-1)
  return { reply: String(lastAI?.content ?? ''), intent: result.intent }
}
```

**工作量**：从 demo 复制图结构 → 加 export 接口函数，约 30 分钟。

### Step 7：修改 `src/routes/chat.ts`（新增路由）

**来源**：在现有 `chat.ts` 末尾追加新路由，**不删除旧路由**。

**新增 3 个路由**：

```typescript
import multer from 'multer'
import { parseFile } from '../services/docParser.js'
import { invokeAgent, resumeAgent } from '../agents/mainGraph.js'
import { fetchTemplatesFromDb } from '../db/mysql.js'

const upload = multer({ dest: 'uploads/', limits: { fileSize: 10 * 1024 * 1024 } })

// 新路由1：统一入口
router.post('/agent', upload.single('file'), async (req, res) => {
  const { message = '', sessionId = 'default' } = req.body

  let documentText = ''
  if (req.file) {
    const ext = path.extname(
      Buffer.from(req.file.originalname, 'latin1').toString('utf8')
    ).toLowerCase()
    documentText = await parseFile(req.file.path, ext)
    // 清理临时文件
    fs.unlink(req.file.path, () => {})
  }

  // 模板数据：优先从请求体获取，否则从 MySQL 查
  let templates = []
  if (req.body.templates) {
    try { templates = JSON.parse(req.body.templates) } catch {}
  }
  if (templates.length === 0) {
    templates = await fetchTemplatesFromDb()
  }

  try {
    const result = await invokeAgent({ userInput: message, documentText, templates, sessionId })
    res.json({ code: 200, data: result })
  } catch (err: any) {
    // interrupt 会抛出特定异常，需要检测并返回给前端
    if (err?.name === 'GraphInterrupt' || err?.message?.includes?.('interrupt')) {
      res.json({ code: 200, data: { interrupted: true, question: String(err.value ?? err.message) } })
    } else {
      res.json({ code: 500, msg: String(err) })
    }
  }
})

// 新路由2：interrupt 恢复
router.post('/agent/resume', async (req, res) => {
  const { sessionId, supplement } = req.body
  const result = await resumeAgent(sessionId, supplement)
  res.json({ code: 200, data: result })
})
```

**旧路由完全保留**：`/chat/send`、`/chat/stream`、`/chat/clear` 不动，让前端可以渐进迁移。

**工作量**：约 30 分钟。

## 七、关键注意事项

### 7.1 interrupt 在 HTTP 中的处理

你的 demo 是 CLI，`interrupt()` 之后手动调用 `Command({ resume })`。在 HTTP 场景中：

```
前端第1次请求 POST /chat/agent
  → 后端 graph.invoke() → 走到 interrupt → 抛出 GraphInterrupt 异常
  → 路由 catch 住，返回 { interrupted: true, question: "缺：奖项等级、时间" }

前端展示提示，用户填写补充信息

前端第2次请求 POST /chat/agent/resume
  → 后端 graph.invoke(new Command({ resume: 补充内容 }), config)
  → 图从断点恢复，继续执行
  → 返回最终结果
```

**关键**：`thread_id` 必须一致，否则找不到断点。前端需要维护 `sessionId` 不变。

### 7.2 MemorySaver 的局限

`MemorySaver` 把状态存在内存里，**服务重启后丢失**。对于你当前的场景（单实例部署）可以接受。如果需要重启后恢复：

- 短期方案：在 `clarifyNode` 被触发时，往 SQLite `conversations` 表存一条标记，重启后可以提示用户"请重新描述需求"
- 长期方案：实现 `SqliteSaver`（继承 `BaseCheckpointSaver`，把 checkpoint 序列化存 SQLite）

### 7.3 和现有前端的兼容

现有前端用的是 `/chat/send` 和 `/chat/stream`。升级后：

- 旧路由不删，前端无需改动
- 新功能（意图分流、interrupt 追问）通过新路由 `/chat/agent` 暴露
- 前端想迁移时，只需要把请求地址从 `/chat/send` 改为 `/chat/agent`，并处理 `interrupted` 状态

### 7.4 `analyzeChain.ts` 的 `generateApplicationRemark` 保留

`/analyze/generate` 路由用的 `generateApplicationRemark()` 是一个简单的一次性生成任务，不需要意图分类、不需要多轮对话，LCEL 完全够用。**不要升级它**。
